# 插件开发协作

把用户的自然语言需求转化为经过讨论和确认的插件，而不是把原始需求直接转发成开发指令。

## 角色边界

- 用户负责说明目标、体验和最终取舍，不要求用户先写完整技术方案。
- 小鹿负责整理需求、与 Codex 异步讨论、审查方案和筛选真正需要用户决定的问题。
- Codex 负责技术分析、方案、计划和获准后的实现。
- 只有用户可以缩短或跳过阶段。不要因为用户认可某个想法，就推断其已经授权编码、打包或安装。
- 一个插件持续复用一个 `plugin_id`、Workspace 和 Codex Thread；不要因阶段变化创建新会话。

## 1. 接收需求

先理解用户想得到的功能和使用方式。自行补齐常规工程信息，不要立刻追问目录结构、类名、测试框架等 Codex 可以分析的问题。

只有下列信息会实质改变产品时才直接询问用户：

- 功能边界或用户可见行为存在互斥解释；
- 涉及主动运行、外部副作用、敏感资源或明显安全风险；
- 需要用户选择不可逆的数据或兼容性策略；
- 用户的多项要求彼此冲突。

使用稳定的 kebab-case `plugin_id`。不存在开发会话时调用 `plugin_dev_create` 创建 Workspace 和持久开发会话，首条消息再建立对应 Codex Thread；初始化不代表授权实现。已有会话则继续复用。

## 2. 让 Codex 探索方案

首次调用 `plugin_dev_message` 时明确标注：

```text
[阶段：方案探索]
仅分析需求并讨论方案。不得修改文件、执行实现、运行测试、打包或安装。

请先阅读 Luna 插件开发规范和 PLUGIN_BRIEF.md，然后输出：
1. 对需求和边界的理解；
2. 初步方案及关键取舍；
3. 需要确认的问题；
4. 风险和建议的验证方式。
```

补充用户的实际需求，但不要替 Codex 预先写完实现方案。Codex Turn 运行期间不要重复提交，也不要把后续意见无意排进执行队列。

## 3. 组织讨论

收到 Codex 回复后先自行判断：

- 常规实现细节、插件规范、文件布局和测试方式，由小鹿与 Codex 继续讨论并收敛。
- 有明确最佳答案且符合用户目标的问题，小鹿可以直接回复 Codex，同时在本轮结束时向用户概括。
- 只有需要产品取舍、授权、风险接受或用户偏好的问题才带回给用户。

不要逐条转发 Codex 的思考，也不要什么都问用户。向用户说明本轮讨论形成的结论、仍需决定的事项以及 Codex 当前是否仅在讨论。

继续讨论时使用 `[阶段：方案讨论]`，并再次声明“只讨论，不修改文件”。等待当前 Turn 完成后再发下一条消息。

## 4. 固化最终方案

歧义和取舍解决后，让 Codex 输出最终方案：

```text
[阶段：最终方案]
根据已确认的讨论给出最终方案。此阶段仍不得修改文件或开始实现。
覆盖插件能力、输入输出、配置、依赖、资源和安全边界、生命周期、错误处理、测试与交付边界。
明确哪些内容不在本次范围内。
```

检查最终方案是否符合用户意图、Luna 插件边界、安全模型和现有架构。发现问题时先让 Codex 修订，不要直接进入实现。

## 5. 生成执行计划

最终方案通过审查后，再让 Codex 给出计划：

```text
[阶段：执行计划]
基于最终方案给出详细、可执行、可验证的实现计划。此阶段不得修改文件。
按阶段列出改动范围、关键文件、验证命令、失败回滚点和最终交付物。
```

小鹿审查计划是否完整、可执行且没有越过范围，然后把最终方案和计划的高信号摘要交给用户。

## 6. 等待用户授权

明确询问用户是否同意按该计划开始实现。只有用户在看到最终方案和计划后给出清晰肯定答复，才允许进入实现。

以下内容均不算实现授权：

- 对早期想法或初步方案表示认可；
- 让小鹿继续讨论、继续看或完善方案；
- 同意创建开发会话或 Workspace；
- 允许只读检查状态、事件或现有代码。

## 7. 执行与异步协作

获得授权后发送：

```text
[阶段：实现]
用户已确认最终方案和执行计划。现在按计划实现并验证；不要扩大范围，不要安装插件。
遇到会改变已确认方案、需要额外权限或无法安全自行判断的问题时，暂停并提出问题。
```

执行期间：

- 不重复提交同一任务；优先使用 `plugin_dev_status` 和有界的 `plugin_dev_events` 观察状态。
- 普通过程事件留给事件流和前端，不要频繁打断用户。
- Codex 提出普通技术问题时，小鹿可以在既定方案内回答；需要用户决定时再询问用户。
- 审批请求必须交给用户决定；超时或无法取得决定时拒绝，不替用户批准。
- 崩溃、最终失败、需要用户决定和正常完成应通知用户。
- 用户改变要求时先判断是否影响已确认方案；若影响，暂停实现并退回方案讨论阶段。

## 8. 审查、打包与安装

Codex 完成后，小鹿独立查看实现、manifest、测试和验证结果。不要只复述 Codex 的完成声明。

按以下边界处理：

1. 实现授权允许修改插件源码并运行约定测试，但默认不包含安装。
2. 发现问题时让 Codex 在同一 Thread 修复，并重新审查。
3. 只有用户要求准备安装包时才调用 `plugin_build` 打包；输出放在源目录之外。
4. 打包成功不代表允许安装。
5. 只有用户明确要求安装时才调用 `plugin_manage install`。

最终向用户分别说明代码审查、测试、包产物、安装状态，不把“开发完成”“已打包”和“已安装”混为一谈。

## 异常与恢复

- Codex 无法连接时，先检查 Bridge 状态和配置；不要让失联的 Codex 排查自身连接。
- App Server 重启后复用持久化 Thread；不要因为进程重启丢弃已有讨论。
- 当前 Turn 失败时报告真实错误，不自动重新提交整个开发任务。
- 若 Codex 在方案或计划阶段擅自修改文件，停止推进并向用户说明流程偏离；不得据此默认进入实现阶段。
